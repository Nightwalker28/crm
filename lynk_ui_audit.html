Sidebar & Navigation
CriticalSidebar.tsxSidebar is 208px wide and cramps the main canvas
w-52 (208px) is too wide for a compact CRM nav. Salesforce/HubSpot use 56–64px icon-rail or collapsible 200px. Right now it permanently eats 208px of every page.
Before
aside class="w-52 flex-col py-6 pl-4"
// Always 208px, no collapse, no icon mode
After
aside class="w-14 hover:w-52 transition-[width]
  group overflow-hidden flex-col py-4"
// 56px collapsed, 208px on hover/expand
// Icons always visible, labels fade in
Add a toggle button. Store collapsed state in localStorage. Show icon-only mode at w-14, expand to w-52. This is the Salesforce/Linear pattern.
▶
CriticalSidebarNav.tsxNoise texture URL loaded from external CDN on every nav render
bg-[url('https://grainy-gradients.vercel.app/noise.svg')] fires a network request on every nav item render. This is a performance and reliability issue.
Before
bg-[url('https://grainy-gradients.vercel.app/noise.svg')]
// External request, CORS risk, CLS flash
After
// In globals.css, add a base64 data URI
// or just remove — the noise adds very
// little at this size and hurts perf
Either inline a tiny noise SVG as a data URI in CSS, or remove it entirely. The visual difference at 14px icon size is imperceptible.
▶
HighSidebarNav.tsxGroup labels "Main" are redundant — remove or make functional
SidebarGroupLabel with just "Main" serves no purpose. Real CRMs use section labels when there are multiple groups (e.g. "CRM", "Finance", "Settings"). One group = no label needed.
Remove the "Main" label. Add meaningful section dividers when you have 3+ distinct nav groups. The Admin collapsible already groups those items correctly.

Layout & Page Structure
CriticalDashboardLayout.tsxDouble-nested rounded border creates a toy "card-in-card" look
The layout wraps children in a rounded-md + border-neutral-800 inner container inside the main rounded outer shell. This creates an awkward double-border look visible on every page. Enterprise apps have ONE frame, not nested frames.
Before
// Outer: bg-neutral-900 + border
// Inner: rounded-md border-neutral-800
//        bg-[#0a0a0a] — nested card
After
// Single content area, no inner border
// main padding: p-0, content: overflow-y-auto
// Pages set their own top padding (px-8 pt-6)
Remove the inner rounded container entirely. Use a single bg-neutral-950 for the main canvas. Pages themselves control their padding. This matches how HubSpot, Linear, and Salesforce do it.
▶
CriticalMultiple pagesPage headers lack visual hierarchy — title, subtitle, and actions all look the same weight
Every page has an h1 (2xl, semibold) with a subtitle and buttons in a flex row. But there's no visual separation between the header zone and the content zone. No top-of-page brand line, no consistent header height.
Before
h1 class="text-2xl font-semibold"
p class="mt-2 text-sm text-neutral-500"
// Sits directly above content, no separation
After
// Sticky page header bar: h-14, border-b
// border-neutral-800, bg-neutral-950/80
// backdrop-blur-sm, px-8
// h1 text-sm font-semibold + breadcrumb
// Actions in header right slot
Create a shared PageHeader component. Fixed height header bar with title + breadcrumb on left, action buttons on right. Content scrolls below it. This is the universal enterprise pattern.
▶
HighMultiple pagesGap between elements is inconsistent — gap-4, gap-5, gap-6 used interchangeably
Page layouts use gap-4, gap-5, gap-6 with no system. Cards use px-5 py-5 or px-4 py-4 inconsistently. This creates a jittery, uneven feel.
Establish a spacing scale: page-level gaps = gap-6, section gaps = gap-4, intra-card gaps = gap-3. Card padding = px-5 py-4 everywhere. Commit and don't deviate.

Buttons & Actions
Criticalbutton.tsx + multiple pagesPrimary buttons use bg-primary which resolves to near-black — invisible hierarchy
In dark mode, --primary is oklch(0.922 0 0) which is near-white, and --primary-foreground is near-black. This creates a jarring white button on a dark background. Compare to HubSpot (orange), Salesforce (blue), Linear (indigo) — all use a brand color for primary actions.
Before
// globals.css dark:
--primary: oklch(0.922 0 0)  // white
// Results in white button, black text
// Looks like an error state, not a CTA
After
// globals.css dark: --primary: oklch(0.55 0.18 240) // brand blue --primary-foreground: oklch(0.98 0 0) // Or use #2c74b3 (your --acumen color!) // You already defined it, just use it
You already have --acumen: #2c74b3 defined in globals.css. Set --primary to your acumen blue in dark mode. This instantly makes the whole app feel intentional and enterprise.

CriticalMultiple page headersToo many button variants on one toolbar — creates visual noise
InsertionOrdersHeader has: ModuleImportExportControls (Import + Export buttons) + Upload button + New Order button = 4 buttons. ContactsHeader has Import + Export + Create = 3. Every header is a different layout.
Before
// 3–4 separate buttons in header

After
// 1 primary action (Create/New) // 1 "..." overflow menu for Import/Export // Consistent on ALL module pages
Use a DropdownMenu for secondary actions (Import, Export). Keep only the primary CTA as a visible button. This is how Salesforce and HubSpot handle it — one clear action, overflow menu for utilities.

Typography & Color
Criticalglobals.cssSplash animation in production — DEV_ALWAYS_SHOW_SPLASH is false but LynkSplash loads on every visit
The ClientLayout shows a 900ms+ loading splash on every page load. Real enterprise SaaS tools (Salesforce, HubSpot) do not show branded splash screens on internal app loads — only on first login. This adds perceived latency.
Remove the splash from dashboard navigation. Keep it only on the initial `/auth/login → /dashboard` redirect. Once authenticated, navigation should be instant with no fade overlay.
▶
Highglobals.css + multiple filesThree font families loaded with no clear role assignment
You load LynkTitle (TTF), Satoshi (via fontshare CDN), and Geist Sans/Mono (Next.js). But most components use font-sans which resolves to Geist. The Satoshi import is wasted bandwidth.
Before
// LynkTitle: logo/brand mark only
// Satoshi: loaded, mostly unused
// Geist: actual UI font (Next.js default)
After
// LynkTitle: font-lynk → logo only // Remove Satoshi fontshare import // Geist Sans: all UI text (--font-geist-sans) // Geist Mono: code/numbers (font-mono)
Remove the Satoshi CDN import from layout.tsx. It's loading 6 font weights for nothing. Geist is a great enterprise font — clean, readable, designed for UI.
▶
HighinsertionOrderDialog.tsx + othersForms use a 2-column grid that breaks badly with many fields
InsertionOrderDialog has 14+ fields in a sm:grid-cols-2 layout. This means users scroll through 7 rows of paired inputs. Enterprise CRMs use sectioned forms with group labels, not a single mega-grid.
Before
// 14 fields in one flat grid
// No sections, no visual grouping
// Customer + Dates + Money all mixed
After
// Section: Customer (name, ref, status) // Section: Dates (issue, effective, due) // Section: Financials (currency, amounts) // Section: Notes // Each section has a small divider label
Add a FieldSet with a FieldLegend (already in your field.tsx!) to group form fields into logical sections. This makes 14-field forms scannable and professional.
▶
PolishCard.tsxNoise texture overlay on every Card is overdone and blurs text slightly
Every Card has an absolute-positioned noise overlay at opacity-10. At small text sizes this can make labels look slightly blurry. The noise is also loaded from an external URL.
Either remove the noise overlay entirely, or replace with a very subtle inset box-shadow border treatment. Modern enterprise apps use flat surfaces with precise border colors — not texture overlays.
